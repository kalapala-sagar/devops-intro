Experiment 1.3 – Pipe and Input / Output Redirection
Aim:
To understand and use pipe, input redirection, and output redirection operators in Linux for combining commands and managing data flow.
________________________________________
Objective:
•	To learn how to redirect input and output using <, >, >>.
•	To understand how to connect multiple commands using | (pipe).
•	To practice real-time examples of command chaining.
•	To understand the importance of redirection in shell scripting.
________________________________________
Software Requirement:
•	Linux Operating System (Ubuntu / Fedora / CentOS)
•	Bash Shell
•	Terminal
________________________________________
Theory:
In Linux, redirection and pipes are used to control where input comes from and where output goes.

1. Input Redirection (<)
Used to provide input to a command from a file.
Example:
sort < names.txt

2. Output Redirection (>)
Redirects command output to a file (overwrites file).
Example:
ls > list.txt

3. Append Output Redirection (>>)
Appends output to an existing file without overwriting.
Example:
echo "New line" >> notes.txt

4. Pipe (|)
Used to connect the output of one command to the input of another.
Example:
ls -l | grep ".txt"

5. Combined Usage Example
cat file.txt | grep "hello" > output.txt

Redirection and pipes allow commands to work together and automate workflow efficiently.
________________________________________
Sample Output:
1. Output Redirection
$ echo "Hello Linux" > demo.txt
$ cat demo.txt
Hello Linux
2. Append Output
$ echo "Second Line" >> demo.txt
$ cat demo.txt
Hello Linux
Second Line
3. Input Redirection
$ sort < names.txt
sagar
kalapala
linus
veda
4. Pipe Example
$ ls -l | grep ".txt"
-rw-r--r-- 1 user user   50 notes.txt
________________________________________

4. What command is used to combine multiple filters?

Experiment 2.2 – File/Directory Ownerships and Permissions

Aim:
To understand and work with file and directory ownerships, permissions, and permission modification commands in Linux.
________________________________________
Objective:
•	To study file ownership (user and group).
•	To learn permission types: read, write, execute.
•	To modify permissions using chmod.
•	To change ownership using chown and chgrp.
•	To understand symbolic and numeric mode representations.
________________________________________

Theory:
Every file and directory in Linux has:
1. Ownership
There are three types of owners:
1.	User (u) – The owner who created the file.
2.	Group (g) – A group of users with shared permissions.
3.	Others (o) – All system users.
Check ownership:
ls -l
Example Output:
-rw-r--r-- 1 user group 120 notes.txt

2. Permissions
Linux supports three basic permissions:
Symbol	Meaning	For Files	For Directories
r	Read	View content	List directory
w	Write	Modify file	Create/Delete inside directory
x	Execute	Run file	Enter directory
Permissions appear as:
rwxr-xr--
(3 sets → user, group, others)

3. Changing Permissions – chmod
a) Symbolic Method
chmod u+x file.txt
chmod g-w file.txt
chmod o+r file.txt
chmod u=rwx,g=rx,o=r file.txt
b) Numeric Method
Permission	Value
r	4
w	2
x	1
Example:
chmod 755 script.sh
chmod 644 notes.txt

4. Changing Ownership – chown
chown user file.txt
chown user:group file.txt

5. Changing Group Ownership – chgrp
chgrp developers project.txt
These commands help secure files and ensure correct access control.
________________________________________
Sample Output:
1. Checking Permissions
$ ls -l report.txt
-rw-r--r-- 1 student student 200 report.txt
2. Changing Permissions
$ chmod 755 script.sh
$ ls -l script.sh
-rwxr-xr-x 1 student student 350 script.sh
3. Adding execute permission
$ chmod u+x test.sh
4. Changing Ownership
$ sudo chown admin:staff notes.txt
5. Changing Group
$ chgrp developers data.txt
________________________________________
Result:
The concepts of file/directory ownership, permissions, and permission modification using chmod, chown, and chgrp were successfully studied and executed.
________________________________________
Viva Questions & Answers:
1. What are three types of Linux file owners?
Ans: User, Group, and Others.
2. What does permission 755 mean?
Ans: User → rwx (7), Group → rx (5), Others → rx (5).
3. What is the difference between chmod and chown?
Ans: chmod changes permissions; chown changes ownership.
4. What does execute permission do for a directory?
Ans: It allows entering/traversing the directory.

Experiment: 4.1 — Hard Links and Symbolic Links
Aim:
To study and understand the creation and working of Hard Links and Symbolic (Soft) Links in a Linux operating system.
________________________________________
Objective:
•	To learn the difference between hard links and symbolic links.
•	To create and verify hard links using the ln command.
•	To create and verify symbolic links using the ln -s command.
•	To understand inode sharing and link behavior when files are removed or modified.
________________________________________
Theory:
Linux stores files using two components:
1.	Inode – Stores metadata and disk block information
2.	Filename – A directory entry pointing to the inode
1. Hard Link
•	A hard link is an additional name for an existing file.
•	It points directly to the same inode as the original file.
•	Deleting the original file does not remove the data as long as a hard link exists.
•	Hard links cannot link directories and cannot cross file systems.
Command to create:
ln original.txt hardlink.txt
2. Symbolic (Soft) Link
•	A symbolic link is like a shortcut.
•	It stores the path of the original file.
•	It points to the filename, not the inode.
•	If the original file is deleted, the symlink becomes broken.
•	Can link files, directories, and across file systems.
Command to create:
ln -s original.txt symlink.txt
Differences Between Hard Link & Soft Link

Feature	Hard Link	Symbolic Link
Points to	inode	file path
Cross-filesystem	No	Yes
Directory linking	No	Yes
If original deleted	Link works	Link breaks
Size	Same as original file	Small (stores path)
________________________________________
Sample Output:
1. Create a file
$ echo "Hello Linux" > file.txt
2. Create a Hard Link
$ ln file.txt hard.txt
3. Create a Soft Link
$ ln -s file.txt soft.txt
4. List with inode numbers
$ ls -li
12345 -rw-r--r-- 2 user user 12 Jan 10 file.txt
12345 -rw-r--r-- 2 user user 12 Jan 10 hard.txt
12346 lrwxrwxrwx 1 user user  8 Jan 10 soft.txt -> file.txt
5. Delete original file
$ rm file.txt
6. Check behavior
$ cat hard.txt

•	close() → closes the file
The file is read character by character and printed to the console.
Output stream is represented by:
•	1 → STDOUT (standard output)
This program behaves similar to:

Algorithm

2.	Read the file name from the command line arguments.
3.	Open the file in read-only mode using open().
4.	Check whether the file is opened successfully:
o	If not opened, display an error message and stop.
5.	Read one character at a time from the file using read().
6.	Write the character to the output stream (screen) using write().
7.	Repeat Steps (5 or 6) until the end of the file (EOF) is reached.
8.	Close the file using close().
9.	Stop the program.

Program (C Code)

#include <stdio.h>
#include <fcntl.h>
#include <unistd.h>

int main(int argc, char *argv[])
{
    int fd;
    char buf;
    int n;

    // Check if filename is provided
    if (argc < 2)
    {
        printf("Usage: %s <filename>\n", argv[0]);
        return 1;

    while ((n = read(fd, &buf, 1)) > 0)

        write(1, &buf, 1);   // 1 means STDOUT
    }

    close(fd); // Close file
    return 0;
}

To simulate ls, we use directory handling functions:
•	opendir() → opens a directory
•	readdir() → reads directory entries

Using stat(), we can identify:

•	Directories
•	Executable files

1.	Start the program.
2.	Read directory name from command line argument.
3.	Open the directory using opendir().
4.	Read each entry using readdir().
5.	For each entry:
o	Check whether it is a file, directory, or executable.
o	Display its name.

________________________________________
Program (C Code)

#include <stdio.h>
#include <dirent.h>
#include <sys/stat.h>

int main(int argc, char *argv[])
{
    DIR *dp;
    struct dirent *entry;
    struct stat fileStat;

    // Check directory argument
    if (argc < 2)
    {
        printf("Usage: %s <directory_name>\n", argv[0]);
        return 1;
    }

    // Open directory
    dp = opendir(argv[1]);

    if (dp == NULL)
    {
        printf("Error: Unable to open directory!\n");
        return 1;
    }

    printf("\nContents of Directory: %s\n", argv[1]);
    printf("---------------------------------\n");

    // Read directory entries
    while ((entry = readdir(dp)) != NULL)
    {
        stat(entry->d_name, &fileStat);

        // Check type of entry
        if (S_ISDIR(fileStat.st_mode))
        {
            printf("[DIR]  %s\n", entry->d_name);
        }
        else if (fileStat.st_mode & S_IXUSR)
        {
            printf("[EXE]  %s\n", entry->d_name);
        }
        else
        {
            printf("[FILE] %s\n", entry->d_name);
        }
    }

    closedir(dp);
    return 0;
}

#include <unistd.h>

{

    char ch;
    int totalLines = 0, currentLine = 0;

A signal handler is a user-defined function that executes automatically when a signal is received.

signal(SIGINT, handler_function);
This allows the program to respond gracefully instead of stopping suddenly.

Experiment: 12.1 One Way Communication Using Pipe
________________________________________
Aim
To write a C program to implement one-way communication between parent and child processes using a pipe.
________________________________________
Objective
•	To understand the concept of Inter Process Communication (IPC).
•	To learn how to use the pipe() system call.
•	To transfer data from parent process to child process using a pipe.
•	To understand the role of fork() in process creation.
________________________________________
Theory
A pipe is an IPC mechanism used for communication between related processes.
•	A pipe provides a unidirectional data channel.
•	Data written at one end can be read from the other end.
Pipe Characteristics:
•	Created using:
pipe(fd);
•	fd[0] → Read end
•	fd[1] → Write end
Process Creation:
•	fork() creates a child process.
•	Parent writes data into the pipe.
•	Child reads data from the pipe.
Thus, communication happens in one direction.


Algorithm
1.	Start the program.
2.	Declare an integer array fd[2] for pipe descriptors.
3.	Create a pipe using pipe(fd).
4.	Create a child process using fork().
5.	Parent process:
o	Close read end fd[0].
o	Write message into fd[1].
6.	Child process:
o	Close write end fd[1].
o	Read message from fd[0].
o	Display the received message.
7.	Stop the program.
________________________________________
Program (C Code)

#include <stdio.h>
#include <unistd.h>
#include <string.h>

int main()
{
    int fd[2];          // Pipe file descriptors
    char write_msg[] = "Hello Kalapala from Parent Process!";
    char read_msg[100];

    // Create pipe
    if (pipe(fd) == -1)
    {
        printf("Pipe creation failed\n");
        return 1;
    }

    // Create child process
    pid_t pid = fork();

    if (pid < 0)
    {
        printf("Fork failed\n");
        return 1;
    }

    // Parent process
    if (pid > 0)
    {
        close(fd[0]); // Close read end

        write(fd[1], write_msg, strlen(write_msg) + 1);
        printf("Parent sent message: %s\n", write_msg);

        close(fd[1]); // Close write end
    }

    // Child process
    else
    {
        close(fd[1]); // Close write end

        read(fd[0], read_msg, sizeof(read_msg));
        printf("Child received message: %s\n", read_msg);

        close(fd[0]); // Close read end
    }

    return 0;
}

A pipe is an IPC mechanism used for communication between related processes.

•	fd[0] → Read end

Experiment: 12.3 Storing Messages in Message Queues (Sender) 
________________________________________
Aim
To write a C program (sender.c) to create a message queue and store (send) three messages in sequential order.
________________________________________
Objective
•	To understand the concept of Message Queues in IPC.
•	To learn how to create a message queue using msgget().
•	To send messages using msgsnd().
•	To store multiple messages with proper sequence order.
________________________________________
Theory
A Message Queue is an IPC mechanism that allows processes to exchange data in the form of messages.
•	Messages are stored in a queue until the receiving process reads them.
•	Communication can occur between unrelated processes.
Important Functions:
•	ftok() → Generates a unique key
•	msgget() → Creates or accesses a message queue
•	msgsnd() → Sends a message to the queue
•	msgrcv() → Receives a message from the queue
•	msgctl() → Controls or deletes the queue
Message Structure:
struct msg_buffer {
    long msg_type;
    char msg_text[100];
};
________________________________________
Algorithm
1.	Start the program.
2.	Generate a unique key using ftok().
3.	Create a message queue using msgget().
4.	Define a message structure.
5.	Repeat 3 times:
o	Read message input from user
o	Assign message type
o	Send message using msgsnd()
6.	Display confirmation for each message sent.
7.	Stop.
________________________________________
Program (sender.c Code)

#include <stdio.h>
#include <sys/ipc.h>
#include <sys/msg.h>
#include <string.h>

// Message structure
struct msg_buffer
{
    long msg_type;
    char msg_text[100];
};

int main()
{
    key_t key;
    int msgid;
    struct msg_buffer message;

    // Generate unique key
    key = ftok("progfile", 65);

    // Create message queue with read and write permissions
    msgid = msgget(key, 0666 | IPC_CREAT);

    printf("Message Queue Created Successfully...\n");

    // Send 3 messages in sequence
    for (int i = 1; i <= 3; i++)
    {
        message.msg_type = i;

        printf("Enter Message %d: ", i);
        fgets(message.msg_text, sizeof(message.msg_text), stdin);

        // Send message to queue
        msgsnd(msgid, &message, sizeof(message.msg_text), 0);

        printf("Message %d Sent Successfully!\n", i);
    }

    return 0;
}

Important System Calls:
•	msgget() → Access message queue
•	msgrcv() → Receive message from queue

Algorithm

#include <sys/ipc.h>
#include <sys/msg.h>

Experiment: 13.1 Sharing Memory Segment Between Processes
________________________________________
Aim
To write a C program to implement Inter Process Communication (IPC) using Shared Memory, where multiple processes can access the same memory segment simultaneously.
________________________________________
Objective
•	To understand the concept of Shared Memory in IPC.
•	To learn how to create a shared memory segment using shmget().
•	To attach shared memory using shmat().
•	To allow communication between processes through a common memory region.
________________________________________
Theory
Shared Memory is the fastest IPC mechanism because processes directly share a common memory area.
Key Points:
•	A shared memory segment is created by one process.
•	Other processes can attach and access the same memory.
•	Multiple users/processes can read/write simultaneously.
Important Functions:
Function	Description
ftok()	Generates a unique key
shmget()	Creates shared memory segment
shmat()	Attaches shared memory to process
shmdt()	Detaches shared memory
shmctl()	Deletes shared memory
________________________________________
Algorithm
1.	Start the program.
2.	Generate a unique key using ftok().
3.	Create shared memory segment using shmget().
4.	Attach the segment using shmat().
5.	Write data into shared memory.
6.	Display the stored data.
7.	Detach shared memory using shmdt().
8.	Remove shared memory using shmctl().
9.	Stop.
________________________________________
Program (C Code)

#include <stdio.h>
#include <sys/ipc.h>
#include <sys/shm.h>
#include <string.h>

int main()
{
    key_t key;
    int shmid;
    char *data;

    // Generate unique key
    key = ftok("shmfile", 65);

    // Create shared memory segment (size 1024 bytes)
    shmid = shmget(key, 1024, 0666 | IPC_CREAT);

    // Attach shared memory to current process
    data = (char *)shmat(shmid, (void *)0, 0);

    printf("Enter data to store in shared memory: ");
    fgets(data, 1024, stdin);

    printf("\nData written in shared memory: %s", data);

    // Detach shared memory
    shmdt(data);

    // Destroy shared memory segment
    shmctl(shmid, IPC_RMID, NULL);

    printf("\nShared memory detached and removed successfully.\n");

    return 0;
}

Experiment: 14.1 Echo Client Server Program using TCP Elementary Functions
________________________________________
Aim
To write a C program to implement an Echo Client-Server communication using TCP socket programming, where the server echoes back the message sent by the client.
________________________________________
Objective
•	To understand TCP based client-server communication.
•	To learn elementary socket functions such as:
socket(), bind(), listen(), accept(), connect(), send(), recv().
•	To implement an Echo Server that sends back the same message received from the client.
________________________________________
Theory
TCP (Transmission Control Protocol): TCP is a connection-oriented protocol that provides reliable communication between client and server.
Echo Client-Server
•	Client sends a message to server.
•	Server receives the message and sends it back (echo).
•	Client displays the echoed message.
Socket Functions Used
Function	Purpose
socket()	Create socket
bind()	Bind socket to IP & port
listen()	Server waits for connections
accept()	Accept client connection
connect()	Client connects to server
send()	Send data
recv()	Receive data
close()	Close socket

Algorithm:

Server Side
1.	Create a socket.
2.	Bind socket to port number.
3.	Listen for client connections.
4.	Accept client request.
5.	Receive message from client.
6.	Send the same message back (echo).
7.	Close connection.
Client Side
1.	Create a socket.
2.	Connect to server using IP and port.
3.	Send message to server.
4.	Receive echoed message from server.
5.	Display output.
6.	Close socket.
________________________________________
Program (C Code)

Echo Server Program (server.c)
#include <stdio.h>
#include <string.h>
#include <unistd.h>
#include <arpa/inet.h>

int main()
{
    int server_socket, client_socket;
    struct sockaddr_in server_addr, client_addr;
    char buffer[1024];
    socklen_t addr_size;

    // Step 1: Create socket
    server_socket = socket(AF_INET, SOCK_STREAM, 0);

    // Step 2: Define server address
    server_addr.sin_family = AF_INET;
    server_addr.sin_port = htons(8080);
    server_addr.sin_addr.s_addr = INADDR_ANY;

    // Step 3: Bind socket
    bind(server_socket, (struct sockaddr*)&server_addr, sizeof(server_addr));

    // Step 4: Listen for connections
    listen(server_socket, 5);
    printf("Echo Server is running...\n");

    // Step 5: Accept client connection
    addr_size = sizeof(client_addr);
    client_socket = accept(server_socket, (struct sockaddr*)&client_addr, &addr_size);

    printf("Client connected!\n");

    // Step 6: Receive message
    recv(client_socket, buffer, sizeof(buffer), 0);
    printf("Received from client: %s\n", buffer);

    // Step 7: Echo back message
    send(client_socket, buffer, strlen(buffer), 0);

    // Step 8: Close sockets
    close(client_socket);
    close(server_socket);

    return 0;
}

    server_addr.sin_family = AF_INET;
    server_addr.sin_port = htons(8080);

    recv(client_socket, buffer, sizeof(buffer), 0);

    // Step 6: Close socket
    close(client_socket);
